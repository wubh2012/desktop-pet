# Desktop Pet Three.js MVP Design

## 背景

本项目第一版目标是做一个 Windows 优先的 Electron 3D 桌面宠物。当前工作目录中已有 `pet.glb`，经元数据检查，该模型是静态 glTF 2.0 模型：

- `nodes=1`
- `meshes=1`
- `skins=0`
- `animations=0`
- `morphPrimitiveCount=0`

这意味着第一版不能直接控制骨骼、手臂、表情或内置动作。第一版采用 Three.js 加载现有 GLB，并通过程序化整体变换模拟待机、点击反馈和移动行为。

## 目标

第一版交付一个可运行的桌宠 MVP：

- Electron 创建透明、无边框、置顶窗口。
- Three.js 加载并显示 `pet.glb`。
- 宠物默认播放程序化待机动画。
- 点击宠物时播放程序化反馈动画。
- 宠物可以在有限范围内模拟移动。
- 用户可以拖动桌宠窗口位置。
- 项目结构为后续接入真实 GLB 动画、骨骼动画、morph target 表情留出接口。

## 非目标

第一版不实现以下内容：

- 单独控制手臂、腿、尾巴等模型部件。
- 表情控制、眨眼、嘴型或情绪系统。
- 语音、聊天、长期记忆、养成数值。
- Live2D、Spine 或其他 2D 动画管线。
- Linux/macOS 完整桌面行为验收。
- 安装包、自动更新、开机自启。

## 平台策略

第一版按 Windows 优先实现和验收。代码避免主动写死 Windows 专属逻辑，但窗口行为只承诺在 Windows 上稳定：

- 透明无边框窗口。
- 窗口置顶。
- 不显示标准浏览器窗口边框。
- 可拖拽移动窗口。

macOS 和 Linux 后续通过平台适配层处理。Linux 的透明窗口、置顶、鼠标穿透受桌面环境和 Wayland/X11 差异影响，不纳入第一版验收。

## 技术方案

### Electron Main

职责：

- 创建主桌宠窗口。
- 配置透明、无边框、置顶、固定初始尺寸。
- 加载本地 renderer 页面或开发服务器页面。
- 管理窗口生命周期，确保关闭后不残留进程。
- 后续可扩展托盘、右键菜单、开机自启。

边界：

- 不直接操作 Three.js 场景。
- 不直接解析 GLB。
- 不保存复杂宠物状态。

### Preload

职责：

- 在 `contextIsolation` 下暴露最小 IPC API。
- 供 renderer 请求窗口拖动、关闭或读取平台信息。

边界：

- 不暴露 Node.js 全量能力。
- 不把文件系统、进程环境、任意 IPC 通道直接开放给 renderer。

### Renderer

职责：

- 初始化 Three.js 渲染器、场景、相机和灯光。
- 使用 `GLTFLoader` 加载 `pet.glb`。
- 对模型执行自动居中和缩放。
- 使用 `Raycaster` 识别鼠标点击是否命中宠物模型。
- 每帧驱动 `PetController` 更新模型变换。

边界：

- 不负责 Electron 窗口创建。
- 不持久化配置。
- 不假设模型一定带动画、骨骼或 morph target。

### PetController

职责：

- 管理第一版宠物状态机。
- 输出程序化变换：`position`、`rotation`、`scale`。
- 实现 `idle`、`walk`、`clicked` 三类行为。
- 后续接入真实 `AnimationMixer` 时保持对外接口稳定。

状态：

- `idle`：默认状态，轻微上下浮动和身体摆动。
- `walk`：在限定范围内左右移动，同时叠加身体摆动和轻微跳动。
- `clicked`：短时弹跳、放大回弹或转动反馈，结束后回到 `idle` 或 `walk`。

边界：

- 不控制不存在的骨骼、表情或局部网格。
- 不直接访问 Electron API。
- 不直接加载模型文件。

## 数据与资产

第一版资产约定：

- 主模型路径：`public/assets/pet.glb` 或同等静态资源目录。
- 启动时加载该模型。
- 若模型加载失败，界面显示开发态错误提示，生产态保留安全降级。

后续可扩展一个模型能力描述结构：

```json
{
  "hasAnimations": false,
  "hasSkeleton": false,
  "hasMorphTargets": false,
  "animationNames": []
}
```

第一版可在加载后根据 GLB 实际内容生成该信息，用于调试和后续功能判断。

## 交互设计

第一版交互保持克制：

- 左键点击模型：触发 `clicked` 动画。
- 拖动窗口：移动桌宠到桌面任意位置。
- 窗口背景透明，不显示应用说明文字。
- 开发模式可以显示轻量调试信息，例如 FPS、模型加载状态；生产模式默认隐藏。

暂不启用全局鼠标穿透。原因是第一版需要稳定点击和拖拽，过早引入穿透会增加窗口命中、拖动和交互判定复杂度。

## 错误处理

- GLB 加载失败：renderer 捕获异常，显示简短错误状态，不泄露本地绝对路径或敏感环境信息。
- WebGL 不可用：显示降级提示，避免白屏。
- Electron 窗口创建失败：主进程记录错误并退出。
- 资源路径错误：开发期直接暴露资源名称，生产期只显示通用错误。

第一版没有网络 I/O。文件读取仅限加载本地静态资源，不实现重试；失败时进入错误状态。

## 测试与验证

自动化测试优先覆盖纯逻辑：

- `PetController` 状态切换。
- `clicked` 动画结束后状态恢复。
- `walk` 不越过限定移动范围。
- 程序化变换输出不产生 `NaN`。

手动验证覆盖桌面集成：

- `npm run dev` 能启动 Electron。
- 背景透明且无窗口边框。
- 宠物置顶显示。
- `pet.glb` 正常加载、居中、尺寸合适。
- 待机动画持续运行。
- 点击宠物有反馈。
- 拖动窗口后位置变化正常。
- 关闭窗口后没有残留 Electron 进程。

## 后续演进

后续如果要实现更接近瑞星狮子的动作，应升级资产管线，而不是继续硬写整体变换：

- 给 GLB 增加骨骼和 `Idle`、`Walk`、`Click` 动画 clips。
- 使用 Three.js `AnimationMixer` 播放和交叉淡入淡出动画。
- 若模型包含 morph targets，再增加表情控制层。
- 如需要强陪伴感和丰富表情，可单独评估 Live2D 版本，但不作为当前 Three.js MVP 的依赖。
